## Ключевые архитектурные решения

Агент - система построенная по принципу ReAct архитектуры. Агент получает данные раз в 15 минут, анализирует их согласно инструкциям в системном промпте, и возвращает результаты своего анализа.

В Агенте используется LLM, которая работает в OpenAI-формате.

Логика Агента реализована в отдельном сервисе и запускается как FastAPI приложение, что обеспечивает интеграцию с любым внешним UI.

## Модули и ответственность
### Интерфейс
Для работы агента был подготовлен симулятор пиццерии.

Симулятор использует реальные обезличенные данные. Источником данных для симулятора выступает база данных.

В симуляторе возможны следующие действия:
1) Запуск и остановка симуляции
2) Регулирование остатков сырья
3) Регулирование метрик - для создания ситуаций, симулирующих повышенный спрос.
4) Получение ответа от агента и подтверждение предложенных им решений

### Агент
ReAct агент с инструментами
Задачи агента
1) Анализировать метрики 
2) Искать узкие места
3) Предлагать решения для нормализации смены
4) Прогнозировать рост заказов

## Workflow

1. Симулятор формирует payload с:
   - timestamp - текущее время сбора метрик
   - metrics - метрики
   - ingredients - информация о доступном сырье
2. `POST /analyze` в `app_run.py` вызывает `Agent.run(metrics)`
3. Агент выполняет цикл LangGraph:
   - системный промпт + payload + информация о погоде + прогноз на 3 часа вперед
   - опциональные вызовы инструментов
   - возврат результатов инструментов обратно в контекст
   - формирование финального ответа
   - формирование действий для менеджера, если они есть
4. API форматирует секции для менеджера и возвращает:
   - `response` - ответ агента о текущем состоянии метрик
   - `buttons` - действия, для которых нужно подтверждение менеджера
5. Симулятор отображает ответ; при нажатии кнопок отправляет `POST /action` - в контекст агента добавляется информация о нажатой кнопке

## Обработка state, memory и context

### Memory и context
Агент хранит информацию о своих действиях в контексте. Размер контекстного окна равен 4 циклам. Цикл - это полный шаг агента: входящие метрики, история вызова инструментов и финальный ответ (полный результат анализа одного запроса). Получается, что в памяти агента хранится история за последние 60 минут работы пиццерии

Когда контекстное окно превышает заданное количество циклов, самый старый цикл в памяти полностью удаляется.

## State
Агент формирует финальный ответ, когда последнее сообщение не содержит `tool_calls`. Если количество инструментов превышает заданное количество, их вызов прекращается и формируется финальный ответ


## Retrieval-контур

Текущий retriever — это операционный доступ к файлам через инструменты:
- графики: `data/schedule.json`, `data/c_schedule.json`
- погода: `data/weather.xlsx`
- остатки по другим точкам: `data/stores.xlsx`
- история: `data/pre_data.xlsx`

Характеристики:
- запросы к таблицам/файлам в tool-функциях;
- доменная фильтрация (по часам/окнам времени/порогам);

Ограничения:
- текущий формат только для MVP. В production-среде работа будет с базой данных

## Инструменты

Доступные инструменты
1) `get_chefs_schedule` - получить график поваров: тех, кто еще не на смене
2) `get_couriers_schedule` - получить график курьеров: тех, кто еще не на смене
3) `get_information_about_items` - получить информацию о доступности сырья в других пиццериях
4) `prepare_stop_item` - подготовить постановку сырья в стоп
5) `call_employee` - подготовить кнопку вызова сотрудника

## Источники данных
Часть данных была сгенерирована для MVP. Поэтому, для простоты использование они хранятся в json/excel файлах.
Список доступных агенту данных
1) График курьеров и поваров
2) Погодные данные
3) Данные о количестве заказов за прошлую неделю в текущий день симуляции 


## Failure modes, fallback и guardrails

Основные failure modes:
- галлюцинации
- отсутствие/повреждение входных файлов данных для инструментов

Текущие fallback-механизмы:
- если ответ агента превышает timeout, возвращается пустой ответ
- если количество вызванных инструментов превышает допустимый порог, дальнейший вызов прерывается и формируется финальный ответ

Текущие guardrails:
- агент не вызывает критичные инструменты - только предлагает их вызвать
- промпт-правила и доменные пороги
- ограничение памяти агента
- ограничения количества вызываемых инструментов за один шаг анализа
