## Agent

## Назначение

Агент управляет полным циклом анализа: подготовкой контекста, переходами между LLM и tools, контролем лимитов и завершением ответа.

Текущая реализация находится в `agent/one_agent.py` и использует LangGraph `StateGraph`

## State machine

Состояния выполнения:

1. `ingest`
   Подготовка входного payload, погоды и опционального прогноза.
2. `agent`
   Вызов LLM с текущим history.
3. `tools`
   Исполнение tool calls, если они были в последнем `AIMessage`.
4. `finalize`
   Возврат финального текста и списка actions.
5. `error`
   Деградация к пустому ответу при исключении.

Фактический граф LangGraph моделирует только цикл `agent <-> tools`, а `ingest/finalize/error` реализованы вокруг него в `run()`.

## Правила переходов

### `ingest -> agent`

Переход происходит всегда после:
- нормализации timestamp
- выборки погодных данных
- расчета прогноза, если `forecast_count % 4 == 0`
- добавления нового `HumanMessage` в историю

### `agent -> tools`

Переход происходит, если:
- последний ответ модели содержит `tool_calls`
- `tool_call_count < max_tool_calls`

### `agent -> finalize`

Переход происходит, если:
- модель вернула обычный текст без `tool_calls`

### `agent -> finalize` по лимиту

Если модель снова хочет вызвать tool, но `tool_call_count >= max_tool_calls`, агент:
- не дает выполнить новый tool
- делает дополнительный LLM вызов без tool binding
- завершает цикл финальным ответом

### `tools -> agent`

После выполнения tools их результаты как `ToolMessage` возвращаются в историю, и LLM получает новый шаг размышлений

### `* -> error`

Любое необработанное исключение в `graph.ainvoke()` переводит запуск в деградированный режим:
- последний пользовательский ввод удаляется из памяти
- пользователю возвращается пустой ответ
- список действий очищается

## Retry и fallback

### Что реализовано сейчас

- Retry для LLM, network и file I/O не реализован.
- Есть только fallback-механизмы:
  - hard timeout `POST /analyze = 90s`;
  - timeout LLM клиента `120s`;
  - принудительное завершение после `max_tool_calls = 10`;
  - пустой ответ при исключении в графе.

### Почему это важно

Система рассчитана на мониторинг near-real-time. Для такого класса задач безопаснее вернуть деградированный ответ, чем удерживать UI в неопределенном состоянии или автоматически исполнять сомнительное действие.

## Stop condition

Запуск считается завершенным, когда выполняется одно из условий:

- модель вернула ответ без `tool_calls`;
- достигнут лимит tool calls и был сформирован plain-text final answer;
- анализ упал с исключением;
- истек hard timeout API-слоя.

## Что считается unsafe answer


В этом MVP ответ считается небезопасным, если агент:

- пытается инициировать важное операционное действие без подтверждения менеджера
- делает уверенные выводы, когда данных недостаточно или они ненадежны
- выходит за пределы разрешенных ему действий
- не может завершить анализ и зацикливается на вызове инструментов

К небезопасным ответам относятся:

- предложение сразу поставить ингредиент в стоп без подтверждения через интерфейс
- предложение сразу вызвать сотрудника без подтверждения менеджера
- утверждения, основанные на отсутствующих, неполных или ошибочных данных
- рекомендации, для которых в системе нет разрешенного инструмента или действия
- ситуация, когда агент продолжает вызывать tools вместо того, чтобы закончить анализ и дать итоговый ответ

Что защищает систему от таких ответов:

- агент может использовать только заранее разрешенный набор tools
- `prepare_stop_item` и `call_employee` не выполняют действие сами, а только подготавливают предложение
- реальное выполнение предполагается только после подтверждения через POST /action
- если tool завершился с ошибкой или по timeout, агент должен перейти к безопасному деградированному ответу
- если во время анализа происходит сбой, система предпочитает вернуть пустой или нейтральный ответ, а не делать рискованное действие

## Известные ограничения
- Нет policy-level classifier для unsafe answers; безопасность обеспечивается структурой tools и confirmation gate, а не отдельной моделью-проверкой
- Нет явного разделения между reasoning state и user-visible explanation
