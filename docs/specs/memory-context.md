## Memory и Context

## Цель

Memory policy нужна для двух задач:
- сохранить краткую причинно-следственную историю последних событий смены
- не допустить бесконтрольного роста latency и token cost

## Текущее состояние памяти

Память агента хранится в `self.messages` и содержит:
- системный промпт
- `HumanMessage` с метриками
- ответы модели
- результаты инструментов
- сообщения о подтвержденных действиях менеджера

## Единица хранения

Основная единица - цикл анализа.

Цикл:
1. новый `HumanMessage` с текущим payload
2. ноль или более `AIMessage` и `ToolMessage`
3. финальный `AIMessage`


## Политика удержания

- Лимит: `max_history_cycles = 4`
- Усечение выполняется в `limit_messages()`
- Системный промпт не удаляется
- Обрезка производится только если текущий запуск еще не вошел в tool loop, то есть при `tool_call_count == 0`

Это защищает текущий цикл от частичной потери контекста

## Context budget

В текущем MVP сontext budget задается не по токенам, а по числу циклов.

В контекст обычно попадают:
- системный промпт
- последние 4 цикла
- новые метрики
- погодный контекст
- прогноз на 3 часа вперед, если текущий цикл кратен 4

Дополнительно:
- tool results добавляются в историю
- подтвержденные действия менеджера тоже попадают в историю
