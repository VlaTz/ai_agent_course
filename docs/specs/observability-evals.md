## Observability / Evals

## Что наблюдаем сейчас

Текущая observability в MVP строится вокруг обычного application logging и локально собираемых артефактов метрик.

Основные сигналы:
- входной payload анализа;
- финальный ответ агента;
- tool calls и их аргументы;
- latency одного запуска;
- token usage LLM вызовов;
- факт деградации по timeout или исключению.

## Логирование

`logger_config.py` настраивает:
- файловый лог `logs/app_YYYYMMDD.log`;
- консольный лог.

Формат записи:
- timestamp;
- logger name;
- level;
- файл и строка;
- сообщение.

Что логируется кодом:
- инициализация агента;
- вызовы tools;
- входной контекст;
- raw response модели;
- предупреждение о превышении лимита tools;
- timeout `/analyze`;
- исключения внутри graph execution.


## Evals и метрики качества

Для MVP настроен сбор следующих метрик

- `tool_accuracy`- насколько верно выбран инструмент под тип проблемы
- `peak_precision` - доля предупреждений о пике, которые подтвердились
- `peak_recall` - доля реальных пиков, которые агент не пропустил
- `stock_accuracy` - насколько точно предсказывается дефицит ингредиентов
- `cost_metrics` - оценка стоимости системы
- `latency_metrics` - время работы системы


## Минимальный eval dataset

Для регулярной проверки системы нужны сценарии:
- нормальная смена без действий
- рост нагрузки на кухне
- рост нагрузки на доставке
- плохая погода без дефицита сотрудников
- дефицит ингредиентов, компенсируемый другой точкой
- критический дефицит с постановкой в стоп
- отсутствие сотрудников в ближайшие 2 часа
