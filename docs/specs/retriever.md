## Retriever

## Назначение

Это набор данных из локальных `json` и `xlsx` источников, которые добавляются либо:
- напрямую в контекст перед запуском LLM
- либо через LLM-tools по требованию

## Источники

| Источник | Где используется | Формат | Назначение                               |
| --- | --- | --- |------------------------------------------|
| `data/pre_data.xlsx` | `Agent.get_forecast()` | Excel | Исторический заказы по часам             |
| `data/weather.xlsx` | `Agent.run()` | Excel | Погодный контекст на ближайшие 3 часа    |
| `data/schedule.json` | `get_forecast()`, `get_chefs_schedule()` | JSON | График поваров                           |
| `data/c_schedule.json` | `get_forecast()`, `get_couriers_schedule()` | JSON | График курьеров                           |
| `data/stores.xlsx` | `get_information_about_items()` | Excel | Остатки ингредиентов по другим пиццериям |

## Retrieval strategy

### 1. Inline retrieval до запуска графа

Перед каждым запуском оркестратора в `Agent.run()` в контекст добавляются:
- входной payload с текущими метриками и остатками
- погодные данные для часов `current_hour .. current_hour + 3`
- прогноз на 3 часа вперед, но только на каждом четвертом цикле (один раз в час)

### 2. On-demand retrieval через tools

Модель может запросить:
- графики сотрудников
- остатки сырья в других пиццериях

Такой retrieval вызывается в инструментах по решению модели

## Погода, history, store data, schedule
History (`pre_data.xlsx`) используется для прогнозного слоя:
  - расчет ожидаемого числа заказов по часам
  - расчет ожидаемой нагрузки на поваров и курьеров
  - расчет ожидаемого расхода ингредиентов на ближайшие 3 часа. 
  - 
Weather (`weather.xlsx`) используется как контекст интерпретации:
  - добавляется в промпт рядом с текущими метриками
  - может объяснить рост времени доставки
  - не изменяет рассчитанный прогноз напрямую

Store data (`stores.xlsx`) не участвуют в baseline-анализе:
  - подгружаются только если модель уже обнаружила риск дефицита
  - используются для формирования рекомендации, а не для вычисления основного прогноза

Schedule data (`schedule.json/c_schedule.json`) не участвуют в baseline-анализе:
  - подгружаются только если модель уже обнаружила проблемы на кухне
  - используются для формирования рекомендации, а не для корректировки метрик

Итого объединение выполняется в два этапа:
1. `metrics + weather + optional forecast` формируют основной контекст анализа
2. `store data` добавляются только по условному tool call
3. `schedule data` добавляются только по условному tool call

## Поведение при неполных данных

### История

- Если в `pre_data.xlsx` нет всех трех часов окна, прогноз строится по доступным строкам.
- Пропущенные часы не интерполируются и не заполняются эвристикой
- Если для часа нет сотрудников на смене, `orders_per_chef` или `orders_per_courier` получают значение `0`

### Погода

- Если в `weather.xlsx` нет части диапазона, в prompt попадает только доступный фрагмент таблицы

### Остатки по другим точкам

- Если `stores.xlsx` не содержит искомый ингредиент, tool вернет пустой DataFrame в строковом виде

### Графики сотрудников

- Если в графике нет сотрудников, доступных в ближайшие 1-2 часа, tool возвращает текст:
  `В ближайшие 2 часа никто на смену не выходит и будет работать только текущий штат`

## Ограничения

- Нет единого retrieval layer: доступ к данным размазан между `Agent.run()`, `get_forecast()` и tool-функциями.
- Нет кеширования чтения `stores.xlsx`; файл читается на каждый tool call - для оперативного управления данными на этапе MVP


## Failure modes

- Файл отсутствует или поврежден
- Формат колонок отличается от ожидаемого
- Временные поля не парсятся в `datetime`
