## Tools и API

## Цель

Tool layer ограничивает действия LLM заранее определенным набором операций. Все чувствительные эффекты проходят через подтверждение менеджера; прямого side effect во внешней операционной системе у модели нет.

## Общие правила

- Инструменты вызываются только из оркестратора LangGraph.
- Tools объявлены как `async`, но их бизнес-логика остается локальной и выполняется в отдельном потоке через `asyncio.to_thread(...)`.
- Для всех tools действует единый per-tool timeout:
  - `tool_timeout_sec = 3.0`
- Для retrieval tools включен retry:
  - `tool_retry_count = 1`
  - итог: до 2 попыток на один вызов.
- Для action tools retry отключен, чтобы не получить дублирующие side effects в памяти.
- При timeout или ошибке tool не валит весь graph, а возвращает нормализованный результат:
  - `TOOL_TIMEOUT: ...`
  - `TOOL_ERROR: ...`
- Чувствительные actions не исполняются автоматически и требуют подтверждения пользователя через UI-кнопку.

## LLM tools

### `get_chefs_schedule(current_hour: str) -> str`

Назначение:
- найти поваров, чья смена начинается через 1-2 часа от указанного времени

Источник: `data/schedule.json`

Timeout:
- per-tool timeout: 3.0 сек на попытку
- retry: 1 повторная попытка

Возможные ошибки:
- некорректный формат `current_hour`, который не парсится через `pandas.to_datetime`
- отсутствует или поврежден `data/schedule.json`
- неожиданный формат JSON

Side effects:
- нет внешних side effects
- пишет запись в лог
- при timeout/ошибке возвращает `TOOL_TIMEOUT` или `TOOL_ERROR` и не меняет состояние системы.

Подтверждение пользователя:
- не требуется

### `get_couriers_schedule(current_hour: str) -> str`

Назначение:
- найти курьеров, чья смена начинается через 1-2 часа

Источник: `data/c_schedule.json`

Timeout:
- per-tool timeout: 3.0 сек на попытку
- retry: 1 повторная попытка
- ограничение сверху идет через общий timeout `/analyze`

Возможные ошибки:
- некорректный `current_hour`
- отсутствует или поврежден `data/c_schedule.json`
- неожиданный JSON schema

Side effects:
- нет
- пишет запись в лог
- при timeout/ошибке возвращает `TOOL_TIMEOUT` или `TOOL_ERROR`

Подтверждение пользователя:
- не требуется

### `get_information_about_items(items_list: list[str]) -> str`

Назначение:
- проверить наличие ингредиентов в других пиццериях и доступность покупки

Источник: `data/stores.xlsx`

Timeout:
- per-tool timeout: 3.0 сек на попытку
- retry: 1 повторная попытка

Возможные ошибки:
- файл `stores.xlsx` не существует или поврежден
- колонки в Excel названы не так, как ожидается
- `items_list` содержит ингредиенты вне доменного словаря

Side effects:
- нет
- пишет запись в лог
- при timeout/ошибке возвращает `TOOL_TIMEOUT` или `TOOL_ERROR`

Подтверждение пользователя:
- не требуется

### `prepare_stop_item(item_name: str) -> str`

Назначение:
- создать предложение поставить ингредиент в стоп

Источник: локальная память процесса; action сериализуется в кнопку

Timeout:
- per-tool timeout: 3.0 сек
- retry отключен

Возможные ошибки:
- невалидное имя ингредиента для `StopItem`
- ошибка сериализации действия в дальнейшем на API-слое

Side effects:
- при успешном выполнении добавляет `StopItem` в глобальный список `actions`
- пишет запись в лог
- не меняет внешние системы и не меняет реальный inventory
- при timeout или ошибке action не создается и в `actions` ничего не добавляется

Подтверждение пользователя:
- обязательно
- после подтверждения UI вызывает `POST /action`, и только тогда в память агента добавляется текст `Ингредиент «... » поставлен в стоп`

### `call_employee(employee_type: str, employee_name: str, time: str) -> str`

Назначение:
- создать предложение вызвать сотрудника раньше начала смены

Timeout:
- per-tool timeout: 3.0 сек
- retry отключен
- ограничение задается общим временем `/analyze`

Возможные ошибки:
- некорректный `employee_type`
- ошибка в содержимом action-объекта

Side effects:
- при успешном выполнении добавляет `CallChef` или `CallCourier` в глобальный список `actions`
- пишет запись в лог
- не делает внешний вызов сотруднику автоматически
- при timeout или ошибке action не создается

Подтверждение пользователя:
- обязательно
- после подтверждения UI вызывает `POST /action`, и только тогда в память агента добавляется текст `Прошу курьера/повара «{name}» выйти пораньше к {time}`


## Guardrails

- Only-allowlisted tools: модель не может вызвать произвольную функцию или внешний API
- Confirmation gate: чувствительные действия требуют подтверждения
- Soft failure: timeout или ошибка отдельного tool не должны автоматически останавливать весь анализ
- No action on failed tool: если `prepare_stop_item` или `call_employee` завершились с `TOOL_TIMEOUT/TOOL_ERROR`, кнопка не создается
- Logging: все tool calls, timeout и ошибки логируются для последующего аудита
